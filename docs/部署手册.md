## 1. 前置条件

### 1.1 本地环境

| 工具 | 版本要求 | 安装方式 |
|------|---------|---------|
| AWS CLI | v2 | `curl "https://awscli.amazonaws.com/awscli-exe-linux-aarch64.zip" -o awscliv2.zip && unzip awscliv2.zip && sudo ./aws/install` |
| AWS CDK | v2 | `npm install -g aws-cdk` |
| Python | 3.12+ | 系统自带或 `sudo apt install python3` |
| Docker | 20+ | 需要运行 Docker daemon（用于构建 ARM64 容器镜像） |
| AgentCore CLI | 最新 | `pip install bedrock-agentcore-cli` |

### 1.2 AWS 配置

```bash
# 确认 AWS 凭证已配置
aws sts get-caller-identity

# 设置默认区域
export CDK_DEFAULT_REGION=us-west-2
export CDK_DEFAULT_ACCOUNT=$(aws sts get-caller-identity --query Account --output text)
```

### 1.3 初始化项目

```bash
git clone <repo-url>
cd sample-host-openclaw-on-amazon-bedrock-agentcore

# 创建 Python 虚拟环境
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

## 2. 配置 cdk.json

部署前需要配置 `cdk.json` 中的关键参数：

```json
{
  "context": {
    "region": "us-west-2",
    "default_model_id": "global.anthropic.claude-opus-4-6-v1",
    "ws_bridge_enabled": true,
    "registration_open": false
  }
}
```

### 关键参数说明

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `region` | `us-west-2` | AWS 部署区域 |
| `default_model_id` | `global.anthropic.claude-opus-4-6-v1` | Bedrock 模型 ID |
| `ws_bridge_enabled` | `true` | 启用多机器人 WebSocket 桥接（钉钉 + 飞书） |
| `ws_bridge_cpu` | `256` | WS Bridge ECS 任务 CPU（256 = 0.25 vCPU） |
| `ws_bridge_memory_mb` | `512` | WS Bridge ECS 任务内存（MB） |
| `registration_open` | `false` | 是否开放注册（false = 仅白名单用户可使用） |
| `session_idle_timeout` | `1800` | 用户会话空闲超时（秒） |

> **重要**：确保 `ws_bridge_enabled` 设置为 `true`，否则不会创建多机器人桥接服务。

## 3. 一键部署

### 3.1 完整部署（首次）

```bash
./scripts/deploy.sh
```

部署分为三个阶段：

| 阶段 | 内容 | 耗时 |
|------|------|------|
| Phase 1 | CDK 基础设施（VPC、安全、AgentCore、监控） | ~5 分钟 |
| Phase 2 | Starter Toolkit（Runtime、ECR、Docker 镜像构建） | ~10 分钟 |
| Phase 3 | CDK 依赖栈（Router Lambda、Cron、**WS Bridge**、Token 监控） | ~5 分钟 |

部署成功后会看到：

```
=== Deploy complete ===

Next steps:
  1. Set up Telegram:          ./scripts/setup-telegram.sh
  2. Set up Slack:             ./scripts/setup-slack.sh
  3. Set up Feishu (webhook):  ./scripts/setup-feishu.sh
  4. Set up DingTalk/Feishu (WS multi-bot): ./scripts/setup-multi-bot.sh
```

### 3.2 部分部署

```bash
./scripts/deploy.sh --phase1         # 仅 Phase 1（CDK 基础设施）
./scripts/deploy.sh --runtime-only   # 仅 Phase 2（Starter Toolkit）
./scripts/deploy.sh --phase3         # 仅 Phase 3（依赖栈，需先完成 Phase 2）
./scripts/deploy.sh --cdk-only       # Phase 1 + Phase 3（跳过 Starter Toolkit）
```

## 4. 配置钉钉机器人

### 4.1 在钉钉开放平台创建应用

1. 访问 https://open-dev.dingtalk.com/
2. 点击「应用开发」→ 创建应用
3. **消息接收模式**：必须选择 **Stream 模式**（不支持 HTTP 模式）
4. 进入「凭证与基础信息」，复制：
   - **ClientId**（AppKey）
   - **ClientSecret**（AppSecret）
5. 进入「权限管理」，添加以下权限：
   - `qyapi_robot_sendmsg` — 发送机器人消息
   - `qyapi_chat_manage` — 管理群聊（用于群消息发送）
   - `qyapi_robot_emotion` — 表情回应（可选，用于显示「🤔思考中」处理状态）
6. 进入「版本管理与发布」→ **发布应用**

> **注意**：应用必须发布后才能接收消息。
> 收到用户消息后，机器人会立即显示「🤔思考中」表情气泡，处理完成后自动撤回。

### 4.2 使用脚本配置

```bash
./scripts/setup-multi-bot.sh add
```

按提示操作：
1. 选择 `1) DingTalk`
2. 输入 Bot ID（默认 `dingtalk-main`，可自定义如 `dingtalk-sales`）
3. 输入 ClientId 和 ClientSecret
4. 脚本自动验证凭证
5. 输入你的钉钉 staffId（如果不知道，跳过此步）
6. 确认重启 ECS 服务

### 4.3 获取 staffId 并添加白名单

如果跳过了白名单步骤：

1. 在钉钉中给机器人发送任意消息
2. 机器人回复拒绝消息，其中包含你的 ID：`dingtalk:01455368xxxxxx`
3. 添加白名单：
   ```bash
   ./scripts/manage-allowlist.sh add dingtalk:01455368xxxxxx
   ```
4. 再次给机器人发消息 — 现在可以正常使用了

### 4.4 添加多个钉钉机器人

重复执行 `./scripts/setup-multi-bot.sh add`，选择不同的 Bot ID：

```bash
# 添加第二个钉钉机器人
./scripts/setup-multi-bot.sh add
# 选择 1) DingTalk
# Bot ID: dingtalk-team-b
# 输入新的 ClientId / ClientSecret
```

同一组织内的多个钉钉机器人，同一用户的 staffId 相同 — 白名单只需添加一次。

## 5. 配置飞书机器人

### 5.1 在飞书开放平台创建应用

1. 访问 https://open.feishu.cn/app
2. 点击「创建自建应用」
3. 进入「凭证与基础信息」，复制：
   - **App ID**
   - **App Secret**

### 5.2 配置权限

进入「权限管理」→ 搜索并开通以下权限：

| 权限 | 说明 | 必须 |
|------|------|------|
| `im:message` | 发送消息 | 是 |
| `im:message.group_at_msg` | 接收群组 @消息 | 是（群聊场景） |
| `im:resource` | 上传/下载图片和文件 | **是（原生文件发送必须）** |
| `im:message:reaction` | 消息表情回应 | 推荐（处理状态指示） |

> **重要**：
> - 缺少 `im:resource` → 文件以链接形式发送，而不是原生文件消息
> - 缺少 `im:message:reaction` → 无法在收到消息时显示「处理中」表情回应（不影响正常功能）

### 5.3 配置事件订阅

进入「事件与回调」→「事件订阅」：

1. **请求方式**：选择「使用长连接（WebSocket）」
2. 点击「添加事件」，搜索并添加：
   - `im.message.receive_v1` — 接收消息

### 5.4 发布应用

进入「版本管理与发布」→ 创建版本 → 申请上线

> 应用审核通过后才能接收消息。企业内部应用通常可以自动审批。

### 5.5 使用脚本配置

```bash
./scripts/setup-multi-bot.sh add
```

按提示操作：
1. 选择 `2) Feishu`
2. 输入 Bot ID（默认 `feishu-main`，可自定义如 `feishu-multi-bot-1`）
3. 输入 App ID 和 App Secret
4. 脚本自动验证凭证
5. 跳过白名单（飞书需要先启动服务才能获取 open_id）
6. 确认重启 ECS 服务

### 5.6 获取 open_id 并添加白名单

1. 在飞书中给机器人发送任意消息
2. 机器人回复拒绝消息，其中包含你的 ID：`feishu:ou_xxxxxxxx`
3. 添加白名单：
   ```bash
   ./scripts/manage-allowlist.sh add feishu:ou_xxxxxxxx
   ```
4. 再次给机器人发消息 — 现在可以正常使用了

### 5.7 飞书 open_id 注意事项

> **重要**：飞书的 `open_id` 是**应用维度**的，同一个用户在不同飞书应用中拥有不同的 `open_id`。
>
> 这意味着：
> - 如果你配置了 2 个飞书机器人，同一用户需要**分别添加白名单**
> - 每个机器人看到的是不同的用户身份，拥有独立的会话和文件空间
> - 这对多租户场景（不同团队使用不同机器人）是合理的

### 5.8 添加多个飞书机器人

```bash
# 添加第二个飞书机器人
./scripts/setup-multi-bot.sh add
# 选择 2) Feishu
# Bot ID: feishu-multi-bot-2
# 输入新的 App ID / App Secret
```

## 6. 添加用户白名单

### 6.1 全局白名单

```bash
# 添加用户
./scripts/manage-allowlist.sh add dingtalk:01455368xxxxxx
./scripts/manage-allowlist.sh add feishu:ou_xxxxxxxx

# 移除用户
./scripts/manage-allowlist.sh remove dingtalk:01455368xxxxxx

# 查看所有白名单用户
./scripts/manage-allowlist.sh list
```

### 6.2 机器人级别白名单（可选）

如果需要限制某个机器人只允许特定用户使用：

```bash
# 为指定机器人添加用户白名单
./scripts/manage-allowlist.sh add-bot feishu-multi-bot-1 feishu:ou_xxxxxxxx

# 移除
./scripts/manage-allowlist.sh remove-bot feishu-multi-bot-1 feishu:ou_xxxxxxxx

# 查看某个机器人的白名单
./scripts/manage-allowlist.sh list-bot feishu-multi-bot-1
```

> 机器人级别白名单是可选的。未设置时，所有全局白名单用户都可以使用该机器人。

### 6.3 用户首次接入流程

当 `registration_open=false` 时，新用户的接入流程：

```
用户给机器人发消息 → 机器人回复拒绝消息（包含用户 ID）
→ 用户将 ID 发给管理员 → 管理员执行 manage-allowlist.sh add
→ 用户再次发消息 → 正常使用
```

## 7. 部署 Admin Console

Admin Console 是一个 Web 管理控制台，提供渠道配置、用户管理、文件浏览等功能，替代 CLI 脚本进行日常管理。

### 7.1 部署后端

Admin Console 的 CDK stack（`OpenClawAdmin`）已包含在 Phase 3 中。如果 Phase 3 已完成，stack 应已部署。否则单独部署：

```bash
source .venv/bin/activate
export CDK_DEFAULT_ACCOUNT=$(aws sts get-caller-identity --query Account --output text)
export CDK_DEFAULT_REGION=us-west-2

cdk deploy OpenClawAdmin --require-approval never
```

部署完成后输出：
```
Outputs:
OpenClawAdmin.AdminApiUrl = https://xxx.execute-api.us-west-2.amazonaws.com/
OpenClawAdmin.AdminClientId = xxxxxxxxxx
OpenClawAdmin.AdminUrl = https://dxxxxxxxxx.cloudfront.net
OpenClawAdmin.AdminUserPoolId = us-west-2_XXXXXXX
```

### 7.2 部署前端

```bash
./scripts/deploy-admin-ui.sh
```

脚本自动完成：
1. 从 CloudFormation 读取 API URL、Cognito 配置
2. 构建 React SPA（Vite + TypeScript）
3. 上传到 S3
4. 刷新 CloudFront 缓存

### 7.3 创建管理员账号

```bash
./scripts/setup-admin.sh your-email@example.com
```

脚本输出临时密码，首次登录时需修改。

### 7.4 登录使用

1. 访问 CloudFront URL（如 `https://dxxxxxxxxx.cloudfront.net`）
2. 使用邮箱 + 临时密码登录
3. 首次登录设置新密码
4. 进入 Dashboard

### 7.5 Admin Console 功能

| 页面 | 功能 | 替代的 CLI 操作 |
|------|------|----------------|
| **Dashboard** | 用户统计、渠道分布、渠道状态 | `aws dynamodb scan` |
| **Channels** | 配置 Telegram/Slack/Feishu/DingTalk 凭证 | `setup-telegram.sh`, `setup-slack.sh`, `setup-feishu.sh` |
| **Channels → Multi-Bot** | 管理 WS Bridge 中的钉钉/飞书 bot | `setup-multi-bot.sh add/remove/list` |
| **Users** | 用户列表、详情、删除；白名单管理 | `manage-allowlist.sh` |
| **Files** | 浏览用户 S3 文件、文本预览、删除 | `aws s3 ls/cp/rm` |

> **注意**：通过 Admin Console 修改 Multi-Bot 配置后，仍需手动重启 ECS 服务使配置生效：
> ```bash
> ./scripts/setup-multi-bot.sh restart
> ```

## 10. 常见问题

### 飞书机器人连接失败

**现象**：日志中出现 `This event loop is already running`

**原因**：飞书 SDK (`lark-oapi`) 使用全局事件循环，所有飞书机器人必须共享同一线程。

**解决**：这已在代码中处理（`BotManager._run_feishu_group()`）。如果仍然出现，检查是否有代码修改破坏了线程模型。

---

### 飞书文件以链接形式发送

**现象**：机器人发送文件时显示为「Click to download」链接，而不是原生文件消息。

**原因**：飞书应用缺少 `im:resource` 权限。

**解决**：
1. 访问 https://open.feishu.cn/app/{你的AppID}/auth
2. 搜索并开通 `im:resource` 权限
3. 重新发布应用版本

---

### 钉钉机器人收不到消息

**检查清单**：
1. 应用是否已发布？（版本管理 → 发布）
2. 消息接收模式是否为 Stream 模式？
3. ClientId / ClientSecret 是否正确？
4. 用户是否在白名单中？

---

### 飞书同一用户需要多次添加白名单

**原因**：飞书 `open_id` 是应用维度的。同一用户在不同飞书应用中拥有不同的 `open_id`。

**解决**：这是飞书平台的设计。每个飞书机器人需要独立添加白名单。

---

### deploy.sh 报错 `runtime_id not set`

**原因**：Phase 2（Starter Toolkit）尚未运行，cdk.json 中缺少 runtime_id。

**解决**：运行完整部署 `./scripts/deploy.sh`，或先运行 `./scripts/deploy.sh --runtime-only`。

---

### ECS 服务启动后立即退出

**检查**：
```bash
# 查看最新日志
aws logs tail /openclaw/ws-bridge --region us-west-2 --since 10m
```

**常见原因**：
- `openclaw/ws-bridge/bots` 密钥未配置或格式错误
- 凭证验证失败（应用未发布、密钥错误）
- 缺少必要的 IAM 权限

## 架构概览

```
  钉钉 / 飞书用户
       │
       ▼ (WebSocket 长连接)
  ┌──────────────────────────┐
  │  WS Bridge (ECS Fargate) │
  │                          │
  │  ┌──────┐  ┌──────────┐ │
  │  │钉钉  │  │飞书 Bot1 │ │
  │  │Bot A │  │飞书 Bot2 │ │
  │  └──┬───┘  └────┬─────┘ │
  │     └─────┬─────┘       │
  │    Shared Core           │
  │  (身份、会话、S3、AI)    │
  └──────────┬───────────────┘
             │
             ▼
  ┌──────────────────────┐
  │  Amazon Bedrock      │
  │  AgentCore Runtime   │
  │  (每用户独立会话)    │
  └──────────────────────┘
```

所有机器人共享同一个 ECS Fargate 任务，通过 Secrets Manager 动态配置。每个用户拥有独立的 AgentCore 会话和 S3 文件空间。

**消息处理流程**：收到用户消息 → 立即显示处理状态（钉钉：🤔思考中表情 / 飞书：OnIt 表情回应） → 调用 AgentCore → 返回回复 → 自动移除处理状态表情。
